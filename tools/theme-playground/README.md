# theme-playground

Local web app for visually re-assigning palette colors to starship prompt segments. Renders the actual `starship prompt` output, lets you click any palette reference and pick a new one, writes the edit straight to `themes/<theme>/starship.toml` on disk.

Sibling of `themes/`. Not stowed (`.stow-local-ignore` excludes `tools/`).

## Running it

```sh
cd tools/theme-playground
bun install            # first time only
bun run dev            # starts both backend (5174) and Vite (5173)
```

Open <http://localhost:5173>. Backend listens on :5174; Vite proxies `/api/*` to it.

Other scripts: `bun test` (slot-discovery unit + golden-fixture tests), `bun run build` (production bundle into `dist/`), `bun run dev:server` / `bun run dev:web` (run the two halves independently).

```
  PaletteStrip.tsx                 22-swatch palette row

test/fixtures/                     golden discovery output per theme (locks behavior across changes)
```

## How a click round-trips

1. User clicks `pick` on a slot row → `PalettePicker` opens
2. User clicks a swatch → `POST /api/themes/<name>/starship` with `{ slotId, newPaletteKey }`
3. Server rereads the file, runs slot discovery, finds the slot by id, splices the new key in (single byte-range edit — preserves comments, whitespace, the `# generated by` header)
4. Server respawns `starship prompt`, captures the new ANSI, returns the full new `AppState`
5. Frontend replaces local state; preview re-renders

If a slot id from a previous fetch is no longer in the server's response (someone else edited the file, or the offsets shifted), the open picker auto-closes and a "slot moved — pick again" toast fires.

## Slot discovery

The interesting bit. `discoverSlots(text, palette, mode)` is a pure function:

- Parses the file with `tree-sitter-toml` (no hand-rolled TOML parsing).
- Walks every `pair` whose value is a string. If the key is style-bearing (`style`, `*_style`, `style_*`), the whole string is a "slot slice". Regardless of key, the contents of every `[label](style)` construct inside the string are also slices.
- Each slice is tokenized with one regex (`(fg:|bg:)?<name>`). Tokens that are starship modifiers (`bold`, `italic`, `dimmed`, …) or names not in the supplied palette are dropped. Bare names default to `fg`, matching starship's own behavior. The lowercase form is used for palette lookup; original case is preserved in the slot's `key` field.

The starship-specific parts (style tokenizer + `[label](style)` extraction) stay hand-rolled because they're not TOML — they're starship's mini-DSL embedded inside TOML strings.

Forward-compat: `mode: "hex-literal"` is stubbed (`throw TODO(v2)`) for the eventual tmux/fzf support, which scans hex literals in generated files instead of palette key names. The collision rule (multiple palette keys often share the same hex) is spelled out in the design doc.

## Font stack

The prompt preview uses `Comic Code, "Hack Nerd Font Mono", Monaco, monospace`, mirroring `.config/ghostty/config`. All three are system-installed (none bundled). The browser falls back glyph by glyph: text from Comic Code, Powerline/Nerd icons from Hack, the apple logo (U+F8FF) from Monaco.
