# Theme playground — design

**Status:** draft (awaiting user review before plan)
**Date:** 2026-05-31
**Related:** [2026-05-30-theming-design.md](2026-05-30-theming-design.md)

> **Execution:** Use `superpowers:subagent-driven-development`. **Do NOT implement in the current agent.** All implementation work must be dispatched to subagents per the plan.

## Why

The theming system (see related spec) generates per-app config from `colors.toml` via templates. Once the palette is correct, the next round of polish is the *prompt aesthetic*: which palette slot each starship segment uses. Today this is a write-template, `theme build`, eyeball, repeat loop — fine for the first pass, painful for iteration.

This playground is a tiny local web tool that loads the active theme's generated `starship.toml`, shows the real `starship prompt` output, and lets you re-assign palette colors to prompt segments with one click. Edits write directly to the rendered file; the preview re-renders against the new file on each save.

## Goals

- See the **actual** `starship prompt` output (not a hand-drawn mock).
- Every palette-reference in the file is a clickable, swatch-pickable slot.
- One click = one round trip = file written + preview re-rendered.
- Zero impact on the rest of the dotfiles repo: the playground lives under its own directory, ignored by Stow, gitignored where appropriate.

## Non-goals (v1)

- Editing tmux.conf / fzf.sh (same pattern is plausible as a follow-up).
- Editing palette hex values (`colors.toml`).
- Editing the template (`themes/templates/starship.toml.tmpl`).
- Adding/removing modules or changing the `format` string structure.
- Multi-theme simultaneous preview, persistent undo, multi-user / hosted.
- Authoring new themes or importing palettes from external sources.

## v1 scope

**App:** starship only.
**Theme:** any directory under `themes/` — default to the active one (`~/.config/themes/current/name`).

## Architecture

### Where it lives & how to run

- **Location:** `tools/theme-playground/` (new top-level directory).
- **Stow:** add `tools` to `.stow-local-ignore`.
- **Runtime:** Bun (already managed via `.config/mise/config.toml`). One process serves the React frontend (via Vite) and a tiny HTTP API for file IO + invoking `starship prompt`.
- **Start:** `cd tools/theme-playground && bun dev`. Optionally `theme playground` as a thin wrapper in `.local/bin/theme`.
- **Repo discovery:** the playground knows its own path; the repo root is `../..`. Override with `--theme-dir`.
- **Theme discovery:** reads `~/.config/themes/current/name` for the default; UI has a dropdown to switch.

### Backend (Bun HTTP server)

Single `server.ts`. Routes:

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/themes` | List directories under `themes/` (excluding `templates/`). Returns names + which is current. |
| `GET` | `/api/themes/:name` | Returns `{ palette, apps: [{ app, fileRaw, colorSlots, preview }] }`. v1 has one entry (`app: "starship"`, `preview: { kind: "ansi", data: <ansi> }`). Forward-compat for tmux/fzf entries (preview kinds `"tmux-bar"`, `"fzf-mock"`). |
| `POST` | `/api/themes/:name/:app` | Body `{ slotId, newPaletteKey }`. Server rewrites the single byte range, writes file, re-renders, returns `{ app, fileRaw, colorSlots, preview }` (or `{ ..., preview: null, error }`). v1 only routes `:app === "starship"`; other values 404. |

**Subprocess:** `Bun.spawn(["starship", "prompt", "--terminal-width=120", "--status=0", "--cmd-duration=1234", "--jobs=0"])` with `cwd = repo root` (so `$git_branch` lights up) and a clean env that includes `STARSHIP_CONFIG`. Captured stdout is the ANSI string.

**TOML reading:** `@iarna/toml` to parse the palette from `colors.toml`. (Bun's built-in TOML support is value-only and doesn't preserve enough surface for us; explicit dep keeps behavior deterministic across runtimes.) **Writing**: not via the AST — we mutate the raw file text by single string-splice at known byte offsets. Preserves the `# generated by` header, comments, ordering, blank lines.

**Concurrency:** single-user localhost; no locking.

### Frontend (React + Vite, plain CSS)

Single page, three regions stacked top to bottom:

```
┌─ header ───────────────────────────────────────────────────┐
│  Theme: [kanagawa ▾]              [reload] [unsaved indicator] │
├─ prompt preview ───────────────────────────────────────────┤
│  ❯ starship prompt output rendered from ANSI (Nerd Font)   │
├─ palette strip ────────────────────────────────────────────┤
│  ■ accent  ■ color0  ■ color1 … ■ color15  ■ background    │
│  (hover → hex + key name; reused by every picker)          │
├─ color-slot table ─────────────────────────────────────────┤
│  Section          Field            Role  Current   →       │
│  ─────────────────────────────────────────────────         │
│  format          (#1)              fg    ■ color12 [pick]  │
│  format          (#2)              fg    ■ color12 [pick]  │
│  format          (#2)              bg    ■ accent  [pick]  │
│  os              style             bg    ■ color12 [pick]  │
│  directory       style             bg    ■ color5  [pick]  │
│  git_branch      format (#1)       bg    ■ color9  [pick]  │
│  …                                                         │
└────────────────────────────────────────────────────────────┘
```

**Components:** `<ThemeSelector>`, `<PromptPreview>` (ANSI → HTML via `ansi_up`, `<pre>` with Nerd Font), `<PaletteStrip>`, `<ColorSlotTable>`, `<PalettePicker>` (popover; clicking commits the change).

**State:** trivial — server is source of truth. After every POST, replace local state with what the server returned. Plain `useState` + a single `useEffect` for initial load. No store library. If a `<PalettePicker>` is open against a `slotId` that's absent from the server's response (concurrent edit, external file edit + `[reload]`, etc.), the picker auto-closes and surfaces a small "slot moved — pick again" toast.

**Styling:** plain CSS.

**Nerd Font:** bundle a single `.woff2` (e.g. JetBrainsMono Nerd Font Regular, ~2 MB) at `tools/theme-playground/public/fonts/`, **checked into the repo**, loaded via `@font-face`. No `setup-fonts.sh`, no copy-from-system. The 2 MB binary is cheaper than the support cost of OS-specific font discovery paths and "I don't have a Nerd Font installed" edge cases.

### Color-slot discovery & write-back

A starship.toml uses palette references in two syntactic places:

1. **Style fields** (single/double-quoted, single-line): `style = "bg:color12 fg:foreground"`.
2. **`[label](style)` constructs** inside any `format` or `*_symbol` field (multi-line allowed via TOML `"""…"""`): `[](fg:color12 bg:accent)`, `[[ $branch ](fg:background bg:color9)]($style)`.

Discovery is a single linear pass producing a list of `ColorSlot` records:

```ts
type ColorSlot = {
  id: string;                  // stable: `${section}/${field}/${role}/${nth}@${start}`
  section: string;             // "[directory]" or "format" for top-level
  field: string;               // "style" or "format (#1)" / "success_symbol" / etc.
  role: "fg" | "bg";
  key: string;                 // "color5", "accent", …
  start: number;               // byte offset of key in the raw file
  end: number;
};
```

**Stage 1 — find style-bearing slices.**

```js
// 1a — single-line `style = "..."` or `*_style = "..."` or `style_* = "..."`
const RE_STYLE_FIELD = /^([ \t]*)([A-Za-z_][A-Za-z0-9_]*)\s*=\s*(["'])((?:(?!\3).)*)\3/gm;
//      Keep matches where field is "style", ends with "_style", OR starts with "style_".
//      The style_* prefix matters: themes/templates/starship.toml.tmpl uses
//      `style_user` and `style_root` in the [username] block; missing these
//      would silently drop palette references on any theme that keeps the block.

// 1b — every [label](style) anywhere in the file (inside any format / symbol field)
const RE_BRACKET_STYLE = /\]\(([^)]*)\)/g;
```

Each match yields a `{ sliceStart, sliceEnd, section, field }`. The section is found by tracking the most recent `^\s*\[([\w.]+)\]\s*$` header up to `sliceStart`. The friendly `field` label for bracket-style matches comes from a backward scan from `sliceStart` to the nearest `^([A-Za-z_]\w*)\s*=` line within the current section — i.e. "this `[](…)` lives inside the `format` field of `[git_branch]`".

**Stage 2 — tokenize each slice and classify.**

```js
const RE_TOKEN = /(fg:|bg:)?([A-Za-z_][A-Za-z0-9_]*)/g;

// Lifted verbatim from starship's parse_style_string (src/config.rs:382-389):
const MODIFIERS = new Set([
  "underline","bold","italic","dimmed","inverted","blink","hidden",
  "strikethrough","prev_fg","prev_bg","none",
]);
```

For each token in a slice:

| Token form | Action |
|---|---|
| `fg:color3` | role `"fg"`, key `color3`, offset = start of `color3` |
| `bg:accent` | role `"bg"`, key `accent` |
| bare `color12` | role `"fg"` (starship treats bare as fg), key `color12` |
| `fg:#83a598` | token regex can't capture `#…` → no match, skipped |
| `bold` / `italic` / `none` / … | matched as bare; rejected by `MODIFIERS` set |
| `$style` (in `]($style)`) | matched as bare `style`; rejected — not in palette keys |

**Critical refinements** (from reading starship source):

- **Lowercase before palette lookup**: `paletteKeys.has(name.toLowerCase())`. Starship lowercases tokens (`token.to_lowercase()` in `parse_style_string`), so `Color12` should resolve to `color12`.
- **Modifier set first**: rejects modifiers before the palette check; clearer semantics, identical outcome.
- **What we deliberately leave alone**: literal `#RRGGBB`, ANSI 0–255 numerics, hardcoded color names (`red`, `bright-blue`, …). Valid starship colors but not palette references. They stay in the file unchanged.

**Validation strategy.** Discovery is verified against committed golden fixtures: every theme under `themes/<name>/starship.toml` has a sibling `tools/theme-playground/test/fixtures/<name>-slots.json` listing the expected slots. CI for the playground re-runs discovery and diffs against the fixtures. This catches both regressions in the discovery code and drift in generated theme files.

Initial verification (2026-05-31) against kanagawa and bamboo covered:

- Powerline pairs in the top-level `format` (e.g. `[](fg:color12 bg:accent)`).
- `style = "…"` rows on every module that declares one.
- `style_user` / `style_root` in `[username]` (bamboo has the block; kanagawa was generated before the username block was added to the template).
- `git_branch.format` and language `*.format` bracketed styles.
- `character.{success,error,vimcmd_*}_symbol` bracketed styles.
- `fg:#83a598` literal hex correctly skipped (regex can't capture `#…` as a name).
- `]($style)` template-var references correctly skipped (`style` is not a palette key).
- `bold` / `italic` / `dimmed` etc. correctly rejected by the modifier set before reaching the palette check.

Slots that go unreferenced in a theme's prompt are expected (those keys are used by tmux/fzf/ghostty, not starship).

**Write-back is a single substring splice:**

```js
const text = await readFile(starshipTomlPath);
const slots = discover(text);             // recomputed every save
const slot = slots.find(s => s.id === id);
const next = text.slice(0, slot.start) + newKey + text.slice(slot.end);
await writeFile(starshipTomlPath, next);
```

Slot offsets are recomputed server-side on every save and shipped back to the frontend; the frontend never holds stale offsets across edits.

**Escape hatch** (if the regex approach ever proves too fragile): port `src/formatter/spec.pest` (~50-line PEG grammar) to a JS PEG via `peggy`. Not needed for v1 — documented so the next maintainer doesn't reinvent it.

### Live re-render flow

```
1. User clicks a swatch in <PalettePicker>
2. Frontend POST /api/themes/:name/starship  { slotId, newPaletteKey }
3. Server:
   a. readFile(starshipTomlPath)
   b. slots = discover(text);  slot = find(slotId)
   c. validate newPaletteKey ∈ the live starship.toml's [palettes.theme] block
      (NOT colors.toml — they can drift if the user edited colors.toml without
      running `theme build`. The picker UI surfaces this drift as a banner.)
   d. text' = splice(text, slot.start, slot.end, newPaletteKey);  writeFile
   e. ansi = spawn(["starship", "prompt", …]).stdout
   f. return { ansi, starshipTomlRaw: text', colorSlots: discover(text') }
4. Frontend replaces local state; <PromptPreview> re-renders.
```

- No SSE, no file watcher, no debouncing.
- Sub-100 ms end-to-end (file write < 1 ms; `starship prompt` ~30–80 ms locally).
- External edits (e.g. `vim themes/kanagawa/starship.toml`): a `[reload]` button in the header re-fetches `/api/themes/:name`.
- Error handling: if `starship prompt` exits non-zero, server returns `{ ansi: null, error: stderr, starshipTomlRaw, colorSlots }`. Frontend shows error inline, keeps previous ANSI, table stays live for recovery.

## Failure modes

| Scenario | Behavior |
|---|---|
| `starship` binary not on PATH | Server returns `{ error: "starship not found" }`; UI shows install hint. |
| `colors.toml` malformed | `/api/themes/:name` returns 500 with parse error; UI shows error and stays on previous theme. |
| `starship.toml` missing or empty | Same — error returned, UI surfaces it. |
| User picks a palette key that exists in UI but was just deleted from `colors.toml` | Server-side validation 400s; UI surfaces; no file write. |
| Multi-line `style = """…"""` (or `*_style`/`style_*`) in the file | Stage 1a regex misses it. We don't generate such files; on theme load, assert no match of `/^\s*([A-Za-z_]\w*)\s*=\s*"""/m` where the captured name is a style-bearing field (`style`, `*_style`, `style_*`). Refuse to load that theme and surface the error in the UI. |
| Comment containing `]( … )` outside triple-quoted strings | Could fake a Stage 1b match. Mitigated by a triple-quote-aware pre-pass: a small state machine walks the file, tracks whether it's inside a `"""…"""` region, and strips `#…` to end-of-line only outside such regions. Called once before Stage 1 runs. |
| External `vim` edit while playground is open | UI state goes stale until `[reload]` is clicked. No watcher in v1. |
| Two simultaneous clicks (same browser, double-click) | Serialize in the Bun handler; second one operates on the freshly-written file. |

## Forward-compatibility (post-v1)

v1 ships starship only, but the design must not paint us into a corner for the next apps the user wants to add — confirmed tmux next; fzf and others plausible. Three constraints to honor in v1 implementation:

**Per-app section layout.** The page is a vertical stack of `[app preview] + [app slot table]` sections. v1 has exactly one section (starship). Adding tmux is a sibling section appended below, not a redesign of v1's chrome. The `<ThemeSelector>` and `<PaletteStrip>` stay shared at the top.

**Slot discovery is `mode`-parameterised, not starship-specific.**

```ts
type SlotMode = "name-token" | "hex-literal";

function discoverSlots(
  text: string,
  palette: Record<string, string>,  // key → "#RRGGBB"
  mode: SlotMode,
): ColorSlot[];
```

- `mode: "name-token"` — starship. Style-slice scan + token regex against palette key names (the v1 algorithm).
- `mode: "hex-literal"` — tmux & fzf (the generated files contain raw hex, because templating already substituted `${palette.background}` → `#1f1f28`). Build a reverse map from the palette, scan the file for `/#[A-Fa-f0-9]{6}/g`, emit a slot for every match whose hex is in the reverse map. Write-back replaces the hex literal with the new key's hex; the file stays valid tmux/fzf syntax.

  **Collision rule.** Multiple palette keys frequently share the same hex (e.g. bamboo's `accent` = `color5` = `color13` = `#aaaaff`; kanagawa's `cursor` = `foreground` = `selection_foreground` = `color15` = `#dcd7ba`). The reverse map must pick a single owner deterministically. The rule, in order: **(1)** prefer the key whose name the *template* uses for that file (read `themes/templates/<app>.tmpl`, walk its `${palette.NAME}` references, and the names that appear become the canonical winners for their hex values); **(2)** if a hex appears in the template under multiple names, prefer the first one (template author's intent); **(3)** if the hex is in the palette but not referenced anywhere in the template, fall back to the first key by `[palette]` declaration order. This makes the reverse-mapping template-aware: editing `tmux.conf`'s background slot maps it back to `background`, not `color0`, because the template wrote `${palette.background}` there.

Same `ColorSlot` shape both modes; same write-back primitive (single byte-range splice). The frontend table component is reused as-is; only the per-app preview component differs.

**Per-app preview renderers are pluggable.**

- starship → ANSI-via-`ansi_up` (v1).
- tmux → CSS state-machine that parses `#[fg=…,bg=…]` markup in `status-left`/`status-right`/`window-status-{,current-}format`, plus a labeled-swatch row for the style-only fields (`pane-border-style`, `mode-style`, `message-style`). No `tmux` subprocess.
- fzf → small CSS mock of a fuzzy-finder pane labelled with the FZF_DEFAULT_OPTS color roles (`bg`, `fg+`, `hl`, `prompt`, `pointer`, `marker`, `spinner`, `header`, `border`). No `fzf` subprocess.

Each renderer is its own component, registered against an app name. Adding "kitty" or "wezterm" later is: write a slot mode (if needed) + a preview component, register it.

**What this changes in v1 code.**

- `slot-discovery.ts` exports `discoverSlots(text, palette, mode)`. v1 only ever calls it with `mode: "name-token"`, but the parameter exists and the `hex-literal` branch may be left unimplemented (clearly marked `TODO(v2)`) or trivially stubbed — implementer's call, just don't hardcode the mode.
- Backend route is `/api/themes/:name/:app` (not `/starship`). v1 only routes `:app === "starship"`; other values 404. Adding tmux is one new case.
- Frontend already structures app-section rendering as a list, not a single section. With one entry in v1.

This is forward-compat, not implementation — none of these add scope to v1; they just prevent v2 from being a rewrite.
